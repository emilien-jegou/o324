Commit 21840d6 (minor): feat: git synchronization, storage refactor, config profiles, CLI
feat: git synchronization, storage refactor, config profiles, CLI

** What is the motivation for the change?

This is a mega-commit that introduce many changes to the CLI

Git storage synchronization:

We can now synchronize the data accross multiple devices, we use a
checkpoint-based approach for reconciliation.

This is the first basic working version of the git storage but many
changes and optimizations will be added in future commits.

Configuration profile:

We can now have multiple profiles in the configuration file, an option
was added to the CLI to select the profile we wish to use.

Better CLI log:

Some changes were added to the CLI to better display the tasks logs, we
can now choose between a simplified, table or json display.

Packages refactors:

We now have three storage packages, they are used as followed

- storage-core: contains the core structure that different storage type
needs to implement.

- storage: interface that should be used by the different bricks of the
applications (*ongoing update*)

- storage-git: first storage implementation

... We also now have a config package which alleviate a lot of issue I had
with cross dependencies.

Commit cf9259b (minor): feat(core): keep an history of all tasks in git storage
feat(core): keep an history of all tasks in git storage

** What is the motivation for the change?

We want to keep an history of every task to allow commands such as logs
and restart in the CLI.

** How does it address the problem?

We keep a list of all logs in the `__metadata.json` file inside the git
storage.

** Are there any side effects?

This will clutter the metadata file with a lot of infos, but that seems
fine, for example, 20 tasks a day for a whole year will use about 0.2 MB
of space.

The real issue could come from either:
- the conversion of the ulid to timestamps
- opening a large amount of json files to get the logs infos
