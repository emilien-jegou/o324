Commit 21840d6 (minor): feat: git synchronization, storage refactor, config profiles, CLI
feat: git synchronization, storage refactor, config profiles, CLI

** What is the motivation for the change?

This is a mega-commit that introduce many changes to the CLI

Git storage synchronization:

We can now synchronize the data accross multiple devices, we use a
checkpoint-based approach for reconciliation.

This is the first basic working version of the git storage but many
changes and optimizations will be added in future commits.

Configuration profile:

We can now have multiple profiles in the configuration file, an option
was added to the CLI to select the profile we wish to use.

Better CLI log:

Some changes were added to the CLI to better display the tasks logs, we
can now choose between a simplified, table or json display.

Packages refactors:

We now have three storage packages, they are used as followed

- storage-core: contains the core structure that different storage type
needs to implement.

- storage: interface that should be used by the different bricks of the
applications (*ongoing update*)

- storage-git: first storage implementation

... We also now have a config package which alleviate a lot of issue I had
with cross dependencies.

Commit 679a110 (minor): feat(CLI): improve log messages formatting
feat(CLI): improve log messages formatting

** What is the motivation for the change?

I want a better way to visualize tasks in the CLI.

** How does it address the problem?

We now have three output options:

A short version that we get by default that omit a lot of informations
tasks (omit ID, task end date ...).

A detailed version when running with `--verbose` that display
informations in a table format.

A json version when running with `--json` that display the tasks as a
json array.

Commit 6c07770 (minor): feat: git storage transaction semaphore implementation
feat: git storage transaction semaphore implementation

** What is the motivation for the change?

We want to be able to lock the access to the storage to ensure updates
are performed atomically.

For the Git storage, this will be done through semaphores, an
implementation was added in this commit.

** Others changes

- We no longer pass the storage directly to handler, instead we give
- Clippy and rustfmt

** Breaking changes

The InMemory storage doesn't have a transaction implementation, using it
will result in a 'todo!' error

Commit ecbb90d (minor): feat: dynamic storage types and start command
feat: dynamic storage types and start command

** What is the motivation for the change?

- I want to be able to modify the storage-type as needed through a CLI
parameter.
- I want the config to be serialized per storage-type so that the user
never have to specify required parameters from storages he doesn't use
- I want users to be able to plug-in their own storage type without
needing to touch the core library internals (although right now they
will still need to modify the CLI for that)

NB: Currently the CLI has direct access to the storage as the
intermediary core layer (command -> core -> storage) is still not
developed.
